## 一、总体结论

当前项目处于"后端 API 骨架基本完整、前端已启动但仅有登录注册页面、核心业务存在多个 P0 阻断问题"的阶段。相比 6/23 评审时，安全方面（密码保护、角色检查、DTO 化）有明显改善，前端从零启动了认证页面和完整 API 层，但文件服务、评分主链路、草稿发布等关键流程仍不可用。

- 后端可编译并在当前基础设施环境中启动，14 个 Controller 覆盖了 API 设计全部模块。
- 多个核心接口仍为 TODO、模拟结果或存在逻辑缺陷。
- 前端从零启动了登录和注册页面，API 层覆盖全部 13 个模块，但课程、作业、资料等业务页面均未实现。
- AI 智能批阅、RAG 答疑和语义相似度分析均依赖 Spring AI，尚未真正集成。
- 文件上传、下载和在线预览仍为全部 stub，未接入 MinIO。
- 后端测试用例数量为 0；前端 `npm run build` 和 `npm run lint` 状态待重新验证。

## 三、关键阻断问题（按优先级排列）

### P0：数据完整性

1. `CourseMember`、`TopicReply`、`Notification` 三个实体有 `deleted` 字段但缺少 `@SQLRestriction("deleted = 0")`——JPA 查询会将已删除的记录连同活跃记录一起返回（`CourseMember.java:27`, `TopicReply.java:24`, `Notification.java:24`）。**注意：** `User`、`Course`、`Assignment`、`Topic`、`LessonDraft`、`Material`、`MaterialFolder` 七个实体已正确添加。
2. `Submission` 实体缺少 `@Table(uniqueConstraints = @UniqueConstraint(name = "uk_assignment_student", columnNames = {"assignment_id", "student_id"}))`（`Submission.java:26`），与数据库设计不一致。

### P0：主流程不可用

3. `SubmissionService.getAssignmentCourseId()` 固定返回 `0L`（`SubmissionService.java:132`），导致教师评分和退回流程无法通过权限检查——教师永远无法批阅任何提交。
4. `FileService` 没有实际上传、下载和预览实现（`FileService.java` 全部 stub），作业附件、提交文件和课程资料链路全部中断。
5. 作业 Controller 丢弃截止时间和附件 ID（`AssignmentController.java:54,57,68` 硬编码 `null`）。
6. 备课草稿发布不会创建目标业务对象（`LessonDraftService.java:60-67`），却会删除草稿，数据永久丢失。
7. `SubmissionService.submit()` 删除旧文件后未关联新文件（`SubmissionService.java:67`），重复提交后提交记录无文件。
8. `ChatController.getHistory()` 调用了 `listSessions()` 返回会话列表而非历史消息（`ChatController.java:54`），API 行为与文档完全不符。

### P0：安全与权限

9. `TopicService.reply()` 回复时 `parentId` 不校验是否属于当前话题（`TopicService.java:88`），可构造跨话题楼中楼。
10. `ChatService.deleteSession()` 没有校验会话所有者（`ChatService.java:90`），任意用户可删除他人 AI 会话。
11. 备课区（`LessonDraftService`）无教师角色限制，学生也可创建/管理草稿。
12. `SubmissionService.listByAssignment()` 无教师权限校验，任意登录用户可查看任意作业的全部提交。
13. `SubmissionService.getDetail()` 无提交者或教师权限校验，任意登录用户可查看任意提交详情。

### P1：构建与交付

14. `npm run build` 和 `npm run lint` 状态待重新验证（上次因 `tsconfig.node.json` project reference 和 ESLint 9 配置问题失败）。
15. Docker Compose 只包含基础设施，不能一键启动完整系统。
16. 后端虽然编译成功，但没有任何自动化测试（0 个测试用例）。

## 五、推荐实现顺序

实现顺序以"先形成安全、可测试的主链路，再增加附加和 AI 能力"为原则。
其中阶段 1b（最小文件能力）是阶段 2 主链路的前置依赖，不能跳过。

### 阶段 1a：修复剩余 P0 安全与数据问题（预计 2-3 天）

目标：消除所有已知越权和数据泄露风险。

1. **补全 `@SQLRestriction`**：为 `CourseMember`（`CourseMember.java:27`）、`TopicReply`（`TopicReply.java:24`）、`Notification`（`Notification.java:24`）添加 `@SQLRestriction("deleted = 0")`。
2. **补全唯一约束注解**：为 `Submission` 添加 `@Table(uniqueConstraints = @UniqueConstraint(name = "uk_assignment_student", columnNames = {"assignment_id", "student_id"}))`。
3. **修复评分主链路**：`SubmissionService` 注入 `AssignmentRepository`，实现 `getAssignmentCourseId()` 正确查询。
4. **修复提交越权**：`SubmissionService.listByAssignment()` 和 `getDetail()` 添加教师/提交者权限校验。
5. **修复聊天越权**：`ChatController.getHistory()` 修正为调用 `getHistory()`；`deleteSession()` 添加所有者校验。
6. **修复跨话题伪造**：`TopicService.reply()` 添加 `parent.topicId == topicId` 校验。
7. **修复备课区权限**：`LessonDraftService` 添加全局 TEACHER 角色检查。
8. **修复匿名数据**：匿名话题和回复的响应中通过 DTO 或 `@JsonIgnore` 屏蔽 `authorId`。
9. **修复发布数据丢失**：`LessonDraftService.publish()` 改为先创建目标实体成功后再删除草稿。

验收标准：所有已知越权路径被封堵；`deleted=1` 的记录不在任何 JPA 查询中出现。

### 阶段 1b：最小文件能力（主链路前置依赖，预计 2-3 天）

目标：使阶段 2 的作业附件和提交文件不再是 TODO。

> ⚠️ 阶段 2 主链路涉及"附件上传"和"文件提交"，两者都依赖真实文件能力。
> 如果跳过此阶段直接做阶段 2，附件和文件部分将保持 TODO，主链路不完整。

1. 设计临时文件实体（`TempFile`：id, url, size, name, uploaderId, expireAt, associated）。
2. 实现 MinIO 上传（大小 ≤ 50MB、扩展名白名单、MIME 校验）。
3. 实现 MinIO 预签名下载 URL（30 分钟有效期）和 PDF/图片预览 URL。
4. 实现"先上传后关联"：临时文件 → 业务实体关联。
5. 定时清理 24 小时内未关联的孤儿文件（`@Scheduled`）。

验收标准：可上传文件并获取真实 URL；可下载；PDF/图片可预览；过期文件自动清理。

### 阶段 2：打通基础业务闭环（预计 5-7 天）

目标：能够演示"注册登录 → 创建/加入课程 → 发布作业 → 提交 → 批阅 → 查看结果"。

1. 完成课程列表 DTO（含课程名、封面、成员数、角色），归档/恢复/退课/删除语义及课程卡片排序。
2. **前端并行**：课程列表页、课程详情页、首页路由守卫。
3. 修复作业 Controller：正确解析 `deadline` 和 `attachmentIds`（`AssignmentController.java:54,57,68`）。
4. 作业状态可见性：学生只能查询 `PUBLISHED` 作业；已发布作业只能修改 `deadline` 和 `content`。
5. 重构 `SubmissionService`：注入 `AssignmentRepository`，正确校验课程成员身份、作业状态、截止时间和重复提交规则；修复文件关联逻辑（`SubmissionService.java:67`）。
6. 实现评分、退回、催交并发送通知（在 `AssignmentService`、`SubmissionService` 中调用 `NotificationService` 创建通知）。
7. **前端并行**：作业详情页、提交页、批阅页。
8. 为上述主链路增加 Service 集成测试和 API 测试。

验收标准：教师和学生可以分别完成全部基础流程，且不同角色不能访问不属于自己的数据。

### 阶段 3：完成附加功能（预计 5-7 天）

目标：达到"良"等级所需的资料、话题和备课能力。

1. 完善资料目录移动、递归删除和跨课程归属校验。
2. 完善话题匿名响应（前端隐藏作者信息）、讨论开关独立接口、回复关系校验和通知。
3. 限制备课区为教师专属（已在阶段 1a 修复）。
4. 实现 `LessonDraftService.publish()`：根据草稿类型反序列化 `contentJson` 调用对应的 `AssignmentService.create()` / `MaterialService.create()` / `TopicService.create()`，成功后删除草稿。
5. **前端并行**：资料管理页、话题讨论页、备课区页、通知中心页。

验收标准：三个附加模块都能独立完成创建、查询、修改、删除和发布操作。

### 阶段 4：实现 AI 智能批阅（预计 4-5 天）

目标：形成第一个真实可演示的 AI 功能。

1. 抽取纯文本、Word 和 PDF 作业内容。
2. 校验 Rubric 权重、满分和 Prompt 模板。
3. 使用 Spring AI `ChatClient` 和结构化输出生成预评分、评语和建议。
4. 对 AI 输出进行范围校验、异常处理和结果持久化。
5. 学生提交后异步触发，教师可重新触发、修改和确认发布。
6. 实现批量任务状态追踪（异步执行 + 进度查询），而不是只返回待处理数量。
7. 增加 Mock 模型测试和至少一组受控真实模型验收用例。

验收标准：相同输入可稳定生成符合 JSON Schema 的结果，失败不会影响正常提交，教师拥有最终决定权。

### 阶段 5：实现课程 RAG 答疑（预计 4-5 天）

目标：基于每门课程独立知识库回答并给出来源。
> 可与阶段 6（相似度）共享 Embedding 基础设施。

1. 建立资料、作业、答案、话题和公告的索引事件。
2. 实现文档解析、分块、Embedding 和 Qdrant 写入/删除。
3. 检索时强制使用 `courseId` 过滤，避免跨课程数据泄露。
4. 拼接检索上下文和受限历史消息，调用 LLM 生成回答。
5. 返回引用名称、片段和来源 ID。
6. 修复 `ChatController.getHistory`（当前调用的是 `listSessions`）；修复会话列表、历史分页和所有者权限。
7. **前端并行**：AI 答疑页面和引用跳转。

验收标准：回答引用真实课程内容，不同课程之间无法相互检索数据，多轮追问可用。

### 阶段 6：实现语义相似度分析（预计 3-4 天）

目标：生成可解释的作业相似度报告。
> 可与阶段 5 共享 Embedding 基础设施，两个模块可并行开发。

1. 提取并标准化提交文本。
2. 批量生成 Embedding 并计算余弦相似度矩阵。
3. 按阈值持久化相似提交对并标记疑似抄袭。
4. 实现相似段落定位和高亮。
5. 增加历史作业批次作为可选比较范围。
6. **前端并行**：相似度报告列表、详情和可视化。
7. 使用人工构造的相同、改写和无关文本建立回归测试。

验收标准：报告包含提交对、分数、阈值判断和可解释高亮，教师可查看且学生不可访问。

### 阶段 7：部署、文档和最终验收（预计 3-4 天）

目标：满足课程提交和答辩要求。

1. 为后端、前端和 Nginx 增加 Dockerfile。
2. 更新 Compose，使前端、后端、MySQL、Redis、MinIO 和 Qdrant 可一键启动。
3. 增加健康检查、依赖顺序、环境变量模板和持久化卷。
4. 更新 README，保证端口和环境变量名称与代码一致。
5. 编写测试报告、部署说明和 AI 辅助编程使用报告。
6. 标注 AI 生成代码，整理关键 Prompt 和人工审查记录。
7. 执行教师、学生、权限、异常、文件、AI 和部署全流程验收。

验收标准：全新环境只需配置必要密钥并执行一次 Compose 命令即可启动，验收用例和提交文档齐全。
